## 致命问题（按优先级）

### P0 · 当前根本跑不起来

- 当前 `.venv` 只装了 **pandas 3.0.3 + numpy 2.5.1**，**sklearn / xgboost / statsmodels / nltk / wordcloud / streamlit / scipy / seaborn / plotly / matplotlib 一个都没装**。这意味着除基础 EDA 外，其余 4 个分析模块 + 仪表板全部导入即失败。
- 项目内**无 `data/`**（需自行从 Kaggle 下载约 1.5GB 的 350 万行 CSV），也**无 `output/` 运行产物**——即文档里声称保存的 `model_comparison.csv` 等文件根本不存在。
- **真实 bug**：`predictive_modeling.py` 里 XGBoost 用了 `use_label_encoder=False`（xgboost≥2.0 已移除，装好后会 `TypeError`）；`LogisticRegression` 用了 `multi_class='multinomial'`（sklearn≥1.5 已弃用）。`requirements.txt` 写 `xgboost>=2.0.0` / `scikit-learn>=1.3.0` 且未封顶，按此安装必然报错。
- **版本号混乱**：`requirements.txt` 写 `pandas>=3.0.0`，但 README 附录写 `pandas>=1.5.0`；README 又说 Python 3.9+，而 pandas 3.0 实际要求 Python≥3.10/3.11。三处对不上。

### P1 · 数据可信度（最致命）

- **报告数字与真实数据集严重不符**：`ANALYSIS_REPORT.md` 称"加州占 98.54%"，但真实 US Accidents 是全国性数据集，加州实际约占 40%~50%，绝无 98.54%；且报告一面说"覆盖美国各地"，一面说加州 98.54%，自相矛盾。
- **样本量自相矛盾**：README 摘要写"约 350 万条、2016-2023"；`ANALYSIS_REPORT` 摘要写"约 35,000 条、2016-2017"；但报告正文表格合计是 50,000 条（CA 49272 + OH 726 + WV 2），且代码默认 `--sample 50000`。摘要、正文、代码三处对不上。
- 月度分布"2 月最低 0.85%"：若是 2016-2023 多年数据，2 月每年都有，不可能仅 0.85%，该数字来源存疑。
- 以上强烈提示报告里的具体数值并非来自真实运行产出，而是手工填入。**对数据分析项目而言，"数字不可信"是致命伤。**

### P2 · 方法学硬伤

- **准确率陷阱**：预测 Severity 准确率 78.52%，但数据集约 60% 是 Severity=2（中等）。若全猜多数类，baseline 就有 ~60%；模型只高出约 18 个点，且加权 F1 掩盖了占比极小的 1 级/4 级（共约 74 条）。报告未给 baseline 对比，也未做类别不平衡处理 / 交叉验证。
- **特征重要性不可比**：对逻辑回归用 `np.abs(coef_).mean(axis=0)` 排序，但温度(0-100)、能见度(0-10)、布尔(0/1) 量纲差异巨大，"温度第一"几乎只是量纲大，结论不可信。
- **大样本滥用 p 值**：卡方 χ²=12847、p<0.001 被当"显著关联"，但 5 万样本下几乎任何微小差异都显著，应报效应量 / 占比差异。
- **天气-严重程度"显著"**：平均严重度 2.386 vs 2.500，差 0.114（1-4 区间内），实际意义极小，却被大书特书。
- 评估函数自写 precision/recall/F1 加权平均（易错、未用 sklearn），README 说有 AUC-ROC 但代码未算。
- 时间序列 `seasonal_decompose(period=365)` 要求至少 2 个周期（>730 天）数据；DBSCAN `eps=1km, min_samples=50` 未调参，直接出"234 个聚类"等数字无法验证。

### P3 · 代码质量

- **重复代码**：`accident_analysis.py` 自己实现了 `load_data` / `preprocess_data`，没复用 `utils/tools.py`；`bool_cols` / `facility_cols` / 中文停用词在三处硬编码相同的大列表（停用词在 `nlp_analysis` + `dashboard` 各写一遍）。
- 风格不统一：`accident_analysis` 不 import utils，其余模块都 import `utils.load_data`。
- 无单元测试、无 CI、无配置管理。

## 所以呢？——改进清单（按性价比排序）

1. **【必做】补齐并锁定依赖**：用真实可装的环境（`pip freeze > requirements.txt` 或 `uv lock`）统一版本；确保 `python run_all_analysis.py --sample 50000` 能从头跑到尾并产出 `output/`。
2. **【必做】修正报告数字**：要么用真实跑出的 `output/` 重新生成所有图表与表格，要么在文档里诚实标注"以下为示例/示意数据"。绝不要在 98.54% 这种与数据集明显冲突的数字上署名。
3. **【必做】修两个会炸的参数**：删 `use_label_encoder=False`；`multi_class` 删掉或锁 sklearn 上限。
4. **【重要】加 baseline 对照**：用 `DummyClassifier(strategy='most_frequent')` 作对比线，所有模型指标都相对它说话；对少数类做 `class_weight` / 重采样，并报告**逐类** precision/recall，而非只报加权。
5. **【重要】特征重要性改用标准化后特征**，或改用树模型 importance / `permutation_importance`；逻辑回归不要用原始 coef 跨量纲比较。
6. **【重要】去掉"大样本 p 值显著"写法**，改报效应量、占比差、置信区间。
7. **【加分】消除重复**：`accident_analysis` 改为 import `utils.load_data` / `preprocess_basic`；将 `FACILITY_COLS` / 中文停用词提取为单一常量模块。
8. **【加分】时序加回测可视化、DBSCAN 参数扫描**；预测目标可考虑改为"是否重大伤以上（Severity≥3）"的二分类，业务意义更清晰。
9. **【加分】加 3~5 个单元测试**（加载、预处理、评估函数），CI 跑通即加分。
